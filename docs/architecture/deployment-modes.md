# Carbon CMS — Deployment Modes

## Overview

Carbon ships in two deployment configurations that share the same codebase and packages:

| Mode | Target | Complexity |
|---|---|---|
| **Solo** | Personal sites, small businesses, non-technical users | One command, zero config |
| **Enterprise** | Multi-server, cloud, distributed teams | Per-component deploys, external services |

Both modes use the same `packages/api`, `packages/admin`, and `packages/frontend` images. The difference is purely in how they are deployed and wired together.

---

## Design Decisions

### Why Docker?

Docker is an implementation detail — users never interact with it directly. The Solo install script checks for Docker and installs it automatically if it is not present, then runs `docker compose up` in the background. From the user's perspective: they ran one command and the site appeared. Docker solves the "works on my machine" problem and makes updates (`docker compose pull && docker compose up -d`) trivially simple.

The Solo install script hides Docker entirely. Advanced users who want to inspect or manage the running containers can do so, but it is never required.

### Why Caddy?

Caddy is included as the required reverse proxy in Solo mode for three reasons:

1. **Automatic HTTPS** — Caddy provisions and renews Let's Encrypt certificates with zero user configuration. The user sets `CARBON_DOMAIN=yourdomain.com` and gets HTTPS for free.
2. **No external port exposure** — Internal services (api, admin, frontend) are never bound to host ports. They communicate only on the Docker internal network. Only ports 80 and 443 are exposed externally. This eliminates a class of exposure risks.
3. **No configuration burden** — Caddy's config is generated by the install script from a template. The user does not write or edit it.

### HTTPS Is Not Optional

HTTPS is enabled by default in all production installs. If `CARBON_DOMAIN` is set, Caddy provisions a Let's Encrypt certificate and all HTTP traffic redirects to HTTPS. There is no unsecured production path.

For local development without a domain, Caddy uses its internal CA to issue a self-signed certificate for `localhost`. Browsers will show a trust warning for local-only installs, which is acceptable for dev.

### Internal Port Randomization

All internal service ports (api, admin, frontend) are assigned random values at install time (range 10000–59999). These ports are never exposed on the host network — they exist only inside Docker's bridge network. Port randomization prevents predictable attack surfaces in cases where someone inadvertently exposes a port.

## Mode 1 — Solo Install

### Target

A non-technical user who wants to run a personal blog or small-business site on a Linux VPS, or use a one-click cloud deploy button. They should not need to understand Docker, reverse proxies, databases, or environment variables.

### Architecture

```
Internet
    │
    ▼ port 443 (HTTPS) / 80 (HTTP → redirect)
┌─────────────────────────────────────────┐
│  Caddy (reverse proxy + TLS termination) │
└────────────┬──────────────┬─────────────┘
             │              │
     /admin* routes    /* routes
             │              │
             ▼              ▼
     ┌─────────────┐  ┌───────────────┐
     │  admin      │  │  frontend     │
     │  :$ADMIN_PORT│  │  :$FRONT_PORT │
     └──────┬──────┘  └──────┬────────┘
            │                │
            │  internal HTTP (Docker bridge)
            │                │
            └────────┬───────┘
                     ▼
             ┌──────────────┐
             │     api      │
             │  :$API_PORT  │
             └──────┬───────┘
                    │
             ┌──────▼───────┐
             │  postgres    │
             │  :5432       │
             └──────────────┘

   Volumes:
     db_data   → postgres data files
     media     → uploaded files (local storage driver)
```

The API is **never reachable from the public internet** in Solo mode. Only Caddy is. Admin and frontend reach the API over Docker's internal bridge network using the Docker service name as hostname.

### URL Structure

```
https://yourdomain.com          →  frontend (public site)
https://admin.yourdomain.com    →  admin UI
```

The admin subdomain is generated automatically by the install script. Both require a single wildcard A record (`*.yourdomain.com → server IP`) or two A records (`yourdomain.com` and `admin.yourdomain.com`). The install script tells the user exactly what DNS records to create.

### Install Experience

The entire install is one command pasted into a terminal:

```bash
curl -fsSL https://get.carbon-cms.io/install.sh | bash
```

Or, for users who download the repo directly:

```bash
./scripts/install.sh
```

The script:

1. **Checks prerequisites** — detects OS (Linux only for self-hosted), checks for `curl`, `openssl`. Installs Docker and Docker Compose automatically if not present (using the official Docker install script for the detected distro).
2. **Collects minimal input** — prompts for:
   - Domain name (e.g. `myblog.com`) — required for HTTPS
   - Email address — required for Let's Encrypt
3. **Generates all secrets** — `AUTH_SECRET` (64-byte hex), `POSTGRES_PASSWORD` (32-byte hex), `INTERNAL_WEBHOOK_SECRET` (32-byte hex), randomized ports for api/admin/frontend.
4. **Writes `.env`** — all generated values, domain, and email. Never prompts for anything else.
5. **Generates `Caddyfile`** from template using the domain and email values.
6. **Runs `docker compose up -d`** — pulls pre-built images from the registry (no build step on the server).
7. **Waits for health checks** — polls `GET /api/v1/health` until ready (max 2 min).
8. **Prints the result**:
   ```
   ✓ Carbon CMS is running.

   Your site:   https://myblog.com
   Admin panel: https://admin.myblog.com

   Complete your setup at https://admin.myblog.com/setup
   to create your first admin account.

   DNS records needed (point both to this server's IP: 1.2.3.4):
     A  myblog.com        → 1.2.3.4
     A  admin.myblog.com  → 1.2.3.4
   ```

### Local Storage for Media

`STORAGE_DRIVER=local` is the default in Solo mode. Uploaded media files are written to a named Docker volume (`media`) mounted into the API container at `$MEDIA_DIR` (default `/app/uploads`). The API serves them at `GET /uploads/:path`.

Media URLs stored in the database use the API's internal URL at write time and are rewritten to the public URL at response time by a `mediaUrl()` helper on the API, using `CARBON_PUBLIC_URL` to construct the correct base.

When a user configures S3 (by setting `AWS_S3_BUCKET`, `AWS_REGION`, etc.), the storage driver switches to S3 automatically and local file serving is bypassed.

### Storage Abstraction Layer

A `StorageDriver` interface is introduced in `packages/api/src/lib/storage.ts`, replacing the current S3-only implementation:

```typescript
interface StorageDriver {
  upload(key: string, buffer: Buffer, mimeType: string): Promise<string>; // returns public URL
  delete(key: string): Promise<void>;
  keyFromUrl(url: string): string;
}
```

Drivers:
- `LocalStorageDriver` — writes to `$MEDIA_DIR`, returns `/uploads/${key}` as the public path
- `S3StorageDriver` — existing behavior, returns `https://${bucket}.s3.../${key}`

The active driver is selected at startup based on `STORAGE_DRIVER`:
```
STORAGE_DRIVER=local   # default for Solo
STORAGE_DRIVER=s3      # for Enterprise or when S3 is configured
```

Vitest unit tests cover both drivers and the driver selection logic.

### Security Model

- Caddy on 443 is the only publicly reachable port.
- Internal service ports are randomized per install and never bound to the host network.
- All inter-service communication is over Docker's internal bridge (no TLS needed — traffic never leaves the host).
- `AUTH_SECRET` is used as the internal admin token between packages. It is generated at install time and stored only in `.env` and the running containers.
- The API's rate limiting middleware applies to all public-facing requests that pass through Caddy.
- `POSTGRES_PASSWORD` is random per install; the Postgres port is never exposed externally.

### docker-compose.yml Changes

The existing `docker-compose.yml` is replaced with a production-ready version:

```yaml
services:
  caddy:
    image: caddy:2-alpine
    ports:
      - "80:80"
      - "443:443"
    volumes:
      - ./Caddyfile:/etc/caddy/Caddyfile:ro
      - caddy_data:/data
      - caddy_config:/config
    depends_on:
      api:
        condition: service_healthy

  db:
    image: postgres:16-alpine
    environment:
      POSTGRES_DB: carbon
      POSTGRES_USER: carbon
      POSTGRES_PASSWORD: ${POSTGRES_PASSWORD}
    volumes:
      - db_data:/var/lib/postgresql/data
    # No host port binding — internal only
    healthcheck:
      test: ["CMD-SHELL", "pg_isready -U carbon"]
      interval: 5s
      timeout: 5s
      retries: 10

  api:
    image: ghcr.io/carbon-cms/api:${CARBON_VERSION:-latest}
    environment:
      DATABASE_URL: postgresql://carbon:${POSTGRES_PASSWORD}@db:5432/carbon
      AUTH_SECRET: ${AUTH_SECRET}
      NEXTAUTH_URL: ${CARBON_API_INTERNAL_URL}
      STORAGE_DRIVER: ${STORAGE_DRIVER:-local}
      MEDIA_DIR: /app/uploads
      CARBON_PUBLIC_URL: https://${CARBON_DOMAIN}
      NODE_ENV: production
    volumes:
      - media:/app/uploads
    # No ports: — internal only
    depends_on:
      db:
        condition: service_healthy
    healthcheck:
      test: ["CMD-SHELL", "wget -qO- http://localhost:${API_PORT}/api/v1/health || exit 1"]
      interval: 10s
      timeout: 5s
      retries: 6

  admin:
    image: ghcr.io/carbon-cms/admin:${CARBON_VERSION:-latest}
    environment:
      CARBON_API_URL: ${CARBON_API_INTERNAL_URL}
      AUTH_SECRET: ${AUTH_SECRET}
      NEXTAUTH_URL: https://admin.${CARBON_DOMAIN}
      NODE_ENV: production
    depends_on:
      - api

  frontend:
    image: ghcr.io/carbon-cms/frontend:${CARBON_VERSION:-latest}
    environment:
      CARBON_API_URL: ${CARBON_API_INTERNAL_URL}
      NEXTAUTH_URL: https://${CARBON_DOMAIN}
      NODE_ENV: production
    depends_on:
      - api

volumes:
  db_data:
  media:
  caddy_data:
  caddy_config:
```

All `PORT` values come from `.env`, written by the install script at random.

### Generated Caddyfile (template)

```
{
  email {$CADDY_EMAIL}
}

{$CARBON_DOMAIN} {
  reverse_proxy frontend:{$FRONTEND_PORT}
}

admin.{$CARBON_DOMAIN} {
  reverse_proxy admin:{$ADMIN_PORT}
}
```

When `CARBON_DOMAIN` is set, Caddy automatically acquires and renews TLS certificates from Let's Encrypt. No additional configuration is needed.

## Mode 2 — Enterprise / Distributed

### Target

Organizations running Carbon at scale: multiple regions, separate infrastructure teams for API vs. frontend, external managed Postgres, S3-compatible object storage, CDN, and the ability to deploy each Carbon component independently.

### Architecture

Each package (`api`, `admin`, `frontend`) is deployed independently. They communicate only through HTTP — there is no shared filesystem, shared process, or shared database access between admin/frontend and the API.

```
                    Internet
                       │
           ┌───────────┼───────────┐
           │           │           │
           ▼           ▼           ▼
    CDN / edge     CDN / edge   CDN / edge
           │           │           │
           ▼           ▼           ▼
   ┌──────────────┐ ┌──────────────┐ ┌──────────────┐
   │  api         │ │  admin       │ │  frontend    │
   │  (any host)  │ │  (any host)  │ │  (any host)  │
   └──────┬───────┘ └──────┬───────┘ └──────┬───────┘
          │                │                │
          │ CARBON_API_URL │ CARBON_API_URL │
          └────────────────┘                │
          ▲                                 │ CARBON_API_URL
          └─────────────────────────────────┘
          │
   ┌──────▼──────────────────┐
   │  Managed Postgres        │
   │  (RDS / Neon / Supabase) │
   └──────────────────────────┘
          │
   ┌──────▼──────────────────┐
   │  S3-compatible storage   │
   │  (AWS S3, R2, MinIO)     │
   └──────────────────────────┘
```

Each host can be a bare VM, a managed container service (Railway, Fly, Render), a Kubernetes pod, or a serverless function. Carbon has no opinion on the host.

### Environment Variables Per Package

**`packages/api`**:
```
DATABASE_URL                Required. Postgres connection string.
AUTH_SECRET                 Required. Shared internal token — must match admin and frontend.
NEXTAUTH_URL                Required. Public URL of this API instance.
STORAGE_DRIVER              Required. "s3" for enterprise.
AWS_S3_BUCKET               Required when STORAGE_DRIVER=s3.
AWS_REGION                  Required when STORAGE_DRIVER=s3.
AWS_ACCESS_KEY_ID           Required when STORAGE_DRIVER=s3.
AWS_SECRET_ACCESS_KEY       Required when STORAGE_DRIVER=s3.
AWS_S3_URL_BASE             Optional. Custom CDN URL for served media.
CARBON_ALLOWED_ORIGINS      Required. Comma-separated list of allowed CORS origins.
                            Example: "https://yourdomain.com,https://admin.yourdomain.com"
INTERNAL_WEBHOOK_SECRET     Required. Signs rebuild webhook payloads to frontend.
```

**`packages/admin`**:
```
CARBON_API_URL              Required. Full URL of the API instance.
AUTH_SECRET                 Required. Must match API.
NEXTAUTH_URL                Required. Public URL of this admin instance.
```

**`packages/frontend`**:
```
CARBON_API_URL              Required. Full URL of the API instance.
NEXTAUTH_URL                Required. Public URL of this frontend instance.
REBUILD_SECRET              Required. Verifies incoming rebuild webhook from API.
```

### What Already Works

- Each package has a standalone Dockerfile and builds independently.
- `CARBON_API_URL` wires admin and frontend to any API URL.
- The API supports external Postgres via `DATABASE_URL` — no Neon dependency.
- The API supports S3 for media storage (existing driver).
- Rate limiting is fully inside the API and works regardless of deployment topology.

## Shared Concerns

### Health Check Endpoint

`GET /api/v1/health` is needed by both modes:

```json
{
  "status": "ok",
  "version": "1.0.0",
  "db": "ok"
}
```

It performs a lightweight DB liveness check (`SELECT 1`) and returns `503` if the database is unreachable. Caddy uses it for health-based routing; docker-compose uses it for `depends_on` conditions; load balancers use it in Enterprise mode.

### Storage Abstraction

The `StorageDriver` interface (see Solo section) is used in both modes. Solo uses `LocalStorageDriver`; Enterprise uses `S3StorageDriver`. No code in the media upload route changes between modes — only the driver selected at startup.

### CORS

The API's CORS middleware reads `CARBON_ALLOWED_ORIGINS` and adds the appropriate `Access-Control-Allow-Origin` response headers. In Solo mode, Caddy handles the routing so admin and frontend appear to browsers as same-origin — CORS is not triggered at all. In Enterprise mode, each service is on its own domain and CORS is required.

In Solo mode, `CARBON_ALLOWED_ORIGINS` does not need to be set (the middleware is a no-op). This keeps the Solo config minimal.

## Decisions (locked)

1. **Published image registry** — GitHub Container Registry (`ghcr.io/carbon-cms/*`). The install script tries `docker compose pull` first; if pre-built images are not yet published it falls back to `docker compose build` automatically. The user sees no difference.

2. **`install.sh` target OS scope** — Linux only for self-hosted (covers all realistic VPS targets). macOS support is for local development via `npm run dev`, not the install script. Windows/WSL2 deferred until there is user demand.

3. **Headless API access in Solo mode** — Off by default. The API is never publicly reachable in a standard Solo install. Users who need headless access (external frontends, mobile apps, API key consumers) set `EXPOSE_API=true` in `.env` and re-run `generate_caddyfile` to add an `api.${CARBON_DOMAIN}` Caddy route. This is documented but never surfaced to non-technical users. An ingress controller can also be placed in front of the internal API port for enterprise network topologies.
